# Deep Audit — Reliability, Performance & Tax-Data Integrity

Date: 2026-06-09. Scope: full sweep of `src/` + `drizzle/`. No code changed.
Context: LAN-only single-user app — security excluded by design; severities calibrated accordingly.

---

## Executive Summary (worst risks first)

The core tax invariant **largely holds**: realized gains and dividends are computed exclusively from
`asset_transactions` → `tax_lots` → `tax_lot_consumptions`, all of which originate from user-entered
or broker-CSV data. No code path in `src/server/tax/` reads `price_history`, and market-data writers
(`price-sync`, both backfills, `setManualPrice`) never touch transaction tables. Tax report generation
makes zero network calls.

However, the invariant is **violated or weakened in five places**:

1. **CRITICAL — Foreign dividends silently converted at FX rate 1.** `createDividend` defaults
   `fxRateToEur ?? 1` and the modal does not require a rate for non-EUR currencies. A $1,000 USD
   dividend recorded without a rate enters the tax report as €1,000 gross — wrong casillas 0027/0029/0588.
2. **HIGH — Silent stale-FX fallback when stamping EUR amounts on transactions.** Both manual entry
   and CSV import fall back to "latest rate ≤ date" (Yahoo/CoinGecko-sourced) with no flag, no audit
   marker. Tax-relevant EUR values can be derived from a rate weeks old with no trace. This also
   bypasses `src/lib/fx.ts` (which *does* track staleness), violating the project's own FX rule.
3. **HIGH — M720/M721/D6 year-end balances silently report €0** for any holding with no valuation
   row, and use arbitrarily stale valuations otherwise — can suppress the €50k/€20k declaration triggers.
4. **HIGH — Import FX prefetch misses dividend/cash rows**, so a non-EUR dividend in a CSV either
   aborts the whole import or silently converts at a stale rate (path 2).
5. **MEDIUM (by design, must stay enumerated) — Binance crypto↔crypto permutas are valued from
   CoinGecko daily closes.** This is the one place market-API data becomes realized-gain proceeds
   and cost basis. Legally defensible (DGT permuta = market value) but unmarked in the data.

Reliability: no timeouts/retries on Yahoo/CoinGecko, cron loops not transactional (a unique-constraint
race aborts a backfill mid-asset), Yahoo quote currency silently defaults to `"USD"`, and two shipped
migrations (`0002`, `0003`) rewrote user account data with no audit trail. Market-API failures cannot
corrupt tax data, but they can block non-EUR transaction entry when `fx_rates` is empty.

Performance: acceptable today; the real items are N+1 valuation lookups across overview/positions,
full-from-epoch valuation rebuilds inside every trade mutation, and per-row inserts outside
transactions in the backfills.

All monetary values are IEEE-754 doubles in `real` columns with ad-hoc rounding; exports can emit
`1234.5600000000002`-style artifacts.

## 1. Tax-Data-Integrity Findings (Priority 1)

### Data-lineage map (report output → originating input)

```
TaxReport.totals.{realizedGains,losses,netComputable,proceeds,costBasis,fees}
 └─ visibleSales[]                                    (server/tax/report.ts:102–180)
     ├─ proceedsEur   = asset_transactions.tradeGrossAmountEur   [sell rows]
     ├─ feesEur       = asset_transactions.feesAmountEur
     ├─ costBasisEur  = Σ tax_lot_consumptions.costBasisEur
     │    └─ tax_lots.unitCostEur = (buy.tradeGrossAmountEur + buy.feesAmountEur) / buy.quantity
     │         └─ buy rows: asset_transactions                   (server/tax/lots.ts:63)
     └─ nonComputableLossEur = Σ tax_wash_sale_adjustments.disallowedLossEur
          └─ derived from the same transaction rows              (server/tax/washSale.ts)

asset_transactions.{tradeGrossAmountEur, feesAmountEur} stamped at entry time by:
 ├─ actions/createTransaction.ts:77–84   qty × priceNative × FX   (user input × FX)
 ├─ actions/confirmImport.ts insertTrade:181–218                  (CSV input × FX)
 ├─ actions/createSwap.ts:36–64          valueEur user-typed, FX=1 (user input)
 └─ actions/createDividend.ts:33–37      grossNative × FX          (user input × FX)
      where FX = explicit user/broker rate → fx_rates[date] → fx_rates[latest ≤ date]
      and fx_rates is populated by Yahoo (`yahoo-fx`) / CoinGecko (`coingecko-fx`)  ← market API

TaxReport.dividends[]    ← asset_transactions dividend rows only            ✓ clean
TaxReport.yearEndBalances[].valueEur ← asset_valuations (Yahoo/CoinGecko/manual)  ← market API
 └─ feeds aggregateBlocksFromBalances → M720/M721/D6 blocks → tax_year_snapshots
```

**Verified-clean paths** (each checked in source):
- Sell proceeds always come from the recorded sell row. A sell cannot be created without an explicit
  positive price: `createTransaction.schema.ts:11` (`priceNative` positive required), DEGIRO/Cobas
  parsers take price/NAV from the CSV itself, swaps require a hand-typed `valueEur`
  (`CreateSwapModal.tsx:197–205` — empty initial value, **no market prefill**).
- No null-coalescing onto market quotes anywhere in the gains path. `grep` for `assetValuations` /
  `priceHistory` / `pricing` across `server/tax/`, `lib/exports/`, `lib/pdf/tax-report.ts`,
  `components/features/taxes/` hits only `report.ts:252` (year-end balances, below).
- Market-data writers write only `price_history`, `fx_rates`, `asset_valuations`
  (`price-sync.ts:143,189,275,286`; `price-backfill.ts`; `setManualPrice.ts`) — never transactions.
- `buildTaxReport` is synchronous and DB-only — fully independent of market-API availability.
- Unrealized vs realized: `asset_valuations`/`asset_positions` never enter `totals.*`; no path
  mistakes a current valuation for a realized gain.

### Findings

**T1 — CRITICAL — Non-EUR dividend silently converted at rate 1.**
`src/actions/createDividend.ts:33` — `const fxRate = data.fxRateToEur ?? 1;`. Schema
(`createDividend.schema.ts`) makes `fxRateToEur` optional with no cross-field rule; the modal
(`CreateDividendModal.tsx:265–275`) computes `needsFx` but does **not** gate submit on it.
A USD dividend without a rate is stored with `grossEur = grossNative`, flowing into casillas
0027/0029 and the DDI computation (0588). Unlike `createTransaction`, there is no `fx_rates`
lookup and no error.
*Fix:* schema `.refine(d => d.currency === "EUR" || d.fxRateToEur != null)`, and/or resolve from
`fx_rates` like `createTransaction` does (erroring when absent); gate the modal's submit on `needsFx`.

**T2 — HIGH — Silent stale-FX fallback stamps tax EUR amounts, bypassing `lib/fx.ts`.**
`src/actions/createTransaction.ts:66–74` and `src/actions/confirmImport.ts:94–103` (`resolveFx`)
fall back to the most recent `fx_rates` row ≤ trade date. The proper helper `lib/fx.ts:resolveFxRate`
returns `{source, stale}` — but neither action uses it, so staleness is invisible: not on the row,
not in the audit event (`createTransaction.ts:169` records the rate value only), not in the UI.
Since `fx_rates` is Yahoo/CoinGecko-sourced, a tax-relevant EUR amount can be derived from a market
rate days or weeks removed from the trade with zero trace. (Also a direct violation of the CLAUDE.md
rule "FX resolution goes through `src/lib/fx.ts`, never ad-hoc".)
*Fix:* route both through `resolveFxRate`; persist `fxSource`/`fxStale` (new column or audit
`contextJson`); show a warning in preview/UI when any row resolved stale.

**T3 — HIGH — Import FX prefetch plan ignores dividend and cash-movement rows.**
`src/actions/confirmImport.ts:49–60` — `gatherFxPlan` only collects currencies from `kind === "trade"`
rows. `insertDividend` (:282) and `insertCashMovement` (:248) still call `resolveFx`. A CSV containing
a USD dividend but no USD trades either (a) aborts the entire import if no USD rates exist (atomic, but
a confusing hard failure), or (b) silently converts at whatever stale rate previous imports left behind
(T2). DEGIRO's parser supplies `fxRateToEurOverride` only when the statement contains a matching FX row.
*Fix:* include all row kinds in `gatherFxPlan`.

**T4 — HIGH — Year-end balances default to €0 when no valuation exists.**
`src/server/tax/report.ts:259` — `const valueEur = valuation ? qty * valuation.unitPriceEur : 0;`.
Unvalued foreign holdings are reported as €0 in M720/M721/D6 blocks. Because the declaration
triggers in `m720.ts:74–79` compare `valueEur` against €50,000 (new) and €20,000 (delta), a missing
valuation can silently suppress a legally required declaration — and the sealed snapshot then
freezes the wrong number.
*Fix:* emit `valueEur: null` + `unvalued: true` and surface prominently in `YearEndCard` and the
M720 export; refuse to seal a year with unvalued foreign balances (or require explicit override).

**T5 — MEDIUM — Year-end valuation staleness unbounded and invisible.**
`src/server/tax/report.ts:252–258` takes the latest `asset_valuations` row ≤ Dec 31 with no recency
bound — could be from months earlier (sync gap, delisted asset). The valuation date is dropped from
`YearEndBalance`, so neither UI nor exports can disclose it.
*Fix:* carry `valuationDate` + `priceSource` into `YearEndBalance`; flag when older than ~10 days
before year-end.

**T6 — MEDIUM (by design — keep enumerated) — Crypto permutas valued from CoinGecko closes.**
`src/lib/imports/binance.ts:151–182` emits the mirror leg of crypto↔crypto trades at
`priceNative = 1` in the quote currency; `insertTrade` converts via `fx_rates`, whose crypto rows are
CoinGecko daily closes (`fx-backfill.ts:30–40, 67–87`). Result: for permutas, **disposal proceeds and
the new lot's cost basis are market-API values**, not user-entered ones, and they use the daily close
rather than execution-time value. Both legs share the same rate (internally consistent), and this
matches DGT V0999-18/V1149-20 treatment — but it is the single point where the invariant is
intentionally relaxed, and nothing in the data marks it.
*Fix:* stamp these rows (e.g. `notes`/`contextJson`: `fx=coingecko-close`), document in SPEC §6, and
disclose in the tax detail export.

**T7 — MEDIUM — Dust filter silently removes disposals from the report.**
`src/server/tax/report.ts:168–179` — sells with `proceedsEur === 0` are always excluded (assumed
import artifacts), and sells with both sides < €1 are dropped. A genuine user-entered micro-sale
vanishes without disclosure; the report carries no count of exclusions, and the sealed snapshot
inherits the omission.
*Fix:* add `excludedSales: {count, proceedsEur, costBasisEur}` to `TaxReport`; render in UI/PDF.

**T8 — MEDIUM — No nominal typing between user-entered money and market valuations.**
`asset_transactions.unitPrice/tradeGrossAmountEur` (user provenance) and
`asset_valuations.unitPriceEur/marketValueEur` (market provenance) are all bare `number`. Nothing
stops a future refactor from passing a valuation where a transaction amount is expected — the
compiler is blind to the distinction the whole app depends on.
*Fix:* branded types at the boundary:
```ts
type TxEur     = number & { readonly __brand: "tx-eur" };      // user/broker-entered
type MarketEur = number & { readonly __brand: "market-eur" };  // quote-derived
```
Constructor functions where rows are read (`src/server/`), `TaxReport` fields typed `TxEur` except
`YearEndBalance.valueEur: MarketEur`. Cheaper interim: ESLint `no-restricted-imports` banning
`asset_valuations`/`pricing` from `src/server/tax/**` except an allow-listed year-end module.

**T9 — MEDIUM — Float arithmetic on money throughout.**
Every monetary column is SQLite `real` (e.g. `db/schema/asset_transactions.ts:19–31`); arithmetic is
IEEE-754 with half-up rounding (`lib/money.ts`). Totals accumulate unrounded
(`report.ts:213–235`), and exports serialize raw doubles: `lib/exports/tax-casillas.ts:24` writes
`${r.valueEur}` (only DDI is rounded), `lib/exports/tax-detail.ts:55` writes `b.valueEur` which is
the unrounded `qty * unitPriceEur` from `report.ts:259`. A casillas CSV can contain
`1234.5600000000002`.
*Fix (minimum):* `roundEur` every total and every exported value. *Fix (proper):* integer cents for
all EUR columns; quantities/FX stay float or move to fixed-point decimal strings.

**T10 — LOW — Global FIFO can misattribute year-end holdings across accounts.**
`server/tax/lots.ts:44–53` replays per asset across **all** accounts (correct Spanish FIFO for
homogeneous values), but `report.ts:240–247` derives per-account year-end quantities from residual
lots keyed by the *buy* account. A sell at broker B drains broker A's oldest lots, so M720 country
blocks can show holdings at the wrong custodian.
*Fix:* compute year-end per-account quantities from per-account transaction sums, not lot residue.

**T11 — LOW — Buys with `quantity ≤ 0` silently skipped in lot replay.**
`server/tax/lots.ts:62` — a corrupted buy row simply vanishes from cost basis. Schemas prevent
creating one, but imported/legacy rows aren't re-validated.
*Fix:* throw (matching the oversell error at :110) or write an audit warning.

**T12 — LOW — Drift detection is totals-only and seal-only.**
`server/tax/seals.ts:38–47` compares six aggregates + two counts; compensating edits (one sale
deleted, another added with equal net) pass undetected, and unsealed years have no drift signal at all.
*Fix:* hash the sorted sale/dividend transaction-id+amount list into the snapshot and compare.

### Where the invariant is assumed but not enforced

| Assumption | Current enforcement | Proposed enforcement point |
|---|---|---|
| Lots/consumptions in sync with transactions | All mutations route through `rebuildAfterTradeMutation` (verified: createTransaction, createSwap, createDividend, deleteTransaction, confirmImport, reimportAccount; deleteAsset/deleteAccount refuse while transactions exist) | Integrity assertion in `buildTaxReport`: Σ`qtyConsumed` per sale == sale qty; error row in report on mismatch |
| Sells always carry a user price | Zod at the action boundary only; DB accepts any `real` | SQLite `CHECK (unit_price > 0)` for buy/sell rows (new migration), + report-time assertion |
| Tax gains never read market tables | Convention only | Lint boundary rule (T8) + regression test R-1 below |
| Broker FX used when the CSV provides it | Parser convention | Fixture test asserting `fxRateToEurOverride` propagates to the stored `fxRateToEur` |
| FX staleness is acceptable | Not tracked at all | Persist `fxSource` (T2) + warn in UI |

## 2. Reliability Findings (Priority 2)

**R1 — HIGH — No timeout, abort, or retry on any external call.**
`src/lib/pricing/yahoo.ts` (both fns), `src/lib/fx-backfill.ts:51` (`yahoo.chart`),
`src/lib/pricing/coingecko.ts`. A hung Yahoo response stalls the cron route (and `confirmImport`'s
FX prefetch — a user-facing flow) indefinitely; transient failures are not retried.
*Fix:* shared wrapper with `AbortSignal.timeout(10_000)` + 2–3 retries with backoff for the cron
paths; import path can stay single-shot (it reports the error cleanly).

**R2 — HIGH — Yahoo quote currency silently defaulted/hardcoded to USD.**
`src/lib/pricing/yahoo.ts:17` (`raw.currency ?? "USD"`) and `:37` (`fetchHistory` hardcodes
`currency = "USD"`). A missing/odd currency from Yahoo silently mislabels the price; `price-sync`
then converts with the wrong FX → corrupt `asset_valuations` (market side; year-end M720 values
affected — tax gains untouched). No cross-check against `asset.currency` anywhere.
*Fix:* throw on missing currency; warn when quote currency ≠ `asset.currency`.

**R3 — HIGH — Backfill/sync write loops are not transactional; concurrent runs abort mid-way.**
`src/lib/price-backfill.ts:137–155` and `:251–291`, `src/lib/price-sync.ts:231–308`. Two overlapping
cron hits both pass the existence check, then one insert hits the unique index and the unhandled
error aborts that run **after** earlier rows were committed — partial backfill, no cleanup. (Single
better-sqlite3 process makes statements atomic, but every `await` is an interleave point between the
read-check and the write.)
*Fix:* wrap per-asset persistence in `db.transaction()` (note: keep fetches outside — no `await`
inside a better-sqlite3 transaction), plus an in-process mutex/`isRunning` flag on the two cron routes.

**R4 — HIGH (pattern) — Shipped migrations rewrote user data with no audit trail.**
`drizzle/0002_cash_bearing_account_types.sql` zeroes `current_cash_balance_eur` and
`opening_balance_eur` for broker/crypto/investment accounts; `0003_prune_account_types.sql`
reclassifies `bank`/`other` → `savings`. Already applied — nothing to fix retroactively, and they
were presumably intentional refactors. Flagged because the *pattern* (silent `UPDATE` of user data
in a migration, no backup, no audit event) is exactly how historical tax inputs get corrupted.
*Fix:* policy + CI check: migrations touching `asset_transactions`/`account_cash_movements`/
`accounts` data (not schema) require a paired audit-event insert and a written rollback note.

**R5 — MEDIUM — Market-API garbage is structurally unvalidated.**
`src/lib/pricing/coingecko.ts` casts `res.json() as T` with no shape check; `fx-backfill.ts:51`
casts the chart result. Confirmed: garbage **cannot reach tax tables** (writers only touch market
tables) and NaN/≤0 closes are filtered (`fx-backfill.ts:58`, `price-sync` checks `quote.price <= 0`),
but malformed shapes surface as confusing deep `TypeError`s and can abort a partially-written sync (R3).
*Fix:* Zod-validate both providers' responses at the client boundary.

**R6 — MEDIUM — Non-EUR manual entry is blocked when `fx_rates` is empty for that currency.**
`src/actions/createTransaction.ts:72` throws `No FX rate available…` — correct refusal (never guess),
but it means first-time entry of a new currency depends on having run an import/sync (market API up).
The schema already accepts an explicit `fxRateToEur`; make sure the transaction modal exposes it as
the offline path, and map this error to a helpful message ("provide the broker's FX rate").

**R7 — MEDIUM — Manual-entry fingerprint collides for legitimate repeat trades.**
`src/actions/_fingerprint.ts` (via `createTransaction.ts:88–95`) hashes account/asset/date/side/
qty/price — two genuinely identical fills on the same day collide; the second insert dies on the
unique index and surfaces as a raw `UNIQUE constraint failed` db error. (Imports solved this with
`rowIndex` — `binance.ts:115–127`.)
*Fix:* include a per-day sequence (or timestamp salt) for `source = "manual"` rows; map the
constraint error to a friendly "possible duplicate" message with an override.

**R8 — MEDIUM — Parse-failed CSV rows are dropped with only a count.**
`confirmImport.ts:470,485` records `skippedErrors: parseResult.errors.length`; row-level messages
exist in `parseResult.errors` but are not persisted or shown post-commit. Missing rows = missing tax
data, discovered only via drift against broker statements.
*Fix:* persist `errors[]` into the import audit event; block commit (or require explicit
acknowledgment) when `errors.length > 0`.

**R9 — MEDIUM — Corrupted seal snapshots are silently treated as absent.**
`server/tax/seals.ts:26–31` and `m720.ts:33–37` swallow `JSON.parse` failures — a sealed year would
silently render live numbers with no "sealed" badge discrepancy warning, and prior-year M720
comparisons would silently lose history.
*Fix:* log + surface a "snapshot unreadable" banner instead of `return null`/`continue`.

**R10 — LOW — Date validation is regex-only.**
All entry schemas (`createTransaction.schema.ts:6–8`, dividend, cash movement) accept impossible
dates (`2025-13-45`); `new Date(...).getTime()` then yields `NaN` for `tradedAt` (NOT NULL column —
insert fails with an opaque error at best). Future dates also accepted.
*Fix:* `.refine` round-trip (`toIsoDate(new Date(d)) === d`) and optionally `d <= today`.

**R11 — LOW — `reimportAccount` is destructive with no export/undo.**
Deletes all transactions/cash movements/lots for the account in one transaction (atomic and
audit-logged, and the UI confirms) — but there's no pre-delete snapshot. Acceptable for LAN; consider
dumping deleted rows into the audit event's `previousJson`.

**R12 — LOW — `price-sync` second run same day skips fresh closes by design.**
`price-sync.ts:103–121` — idempotency is implemented as "skip if any row exists today", so a re-run
after market close doesn't refresh. Document or switch to upsert.

## 3. Performance Findings (Priority 3)

*(Single-user SQLite — none of these hurt today; they're growth debt. Highest-value first.)*

**P1 — MEDIUM-HIGH — Full-from-epoch valuation rebuild inside every trade mutation.**
`server/mutations.ts:27–40` → `server/valuations.ts:49–135`: every `createTransaction`/`delete`/
import wipes and rebuilds the asset's entire daily valuation series (days × trades inner loop,
`valuations.ts:110–117` rescans all trades per day) synchronously inside the write transaction.
A 10-year asset ≈ 2,600 day-rows recomputed per keystroke-level mutation.
*Fix:* incremental rebuild from the earliest affected trade date; replace the per-day trade rescan
with a cursor (as `price-backfill.ts:251–258` already does).

**P2 — MEDIUM — N+1 query patterns on hot read paths.**
- `server/positions.ts:34–54` — latest valuation per position, one query each.
- `server/assets.ts:30–58` — latest price per asset (`Promise.all` of N queries).
- `server/overview.ts:128–142, 288–307, 513–529` — per-asset valuation lookups in loops.
- `server/tax/report.ts:117–166, 249–258` — per-sale consumption/lot/asset queries and per-balance
  account/asset/valuation queries.
*Fix:* batch with `IN (...)` + window function (`ROW_NUMBER() OVER (PARTITION BY ...)`) or pre-fetch
maps. Report builder: fetch all consumptions/lots/assets for the year in three queries.

**P3 — MEDIUM — Bulk writes not wrapped in transactions.**
`price-backfill.ts:137–155, 263–291`, `price-sync.ts:231–308` — per-row `.run()` autocommits;
better-sqlite3 batch inserts inside one transaction are ~100×. Same change as R3 fixes both.

**P4 — MEDIUM — Missing indexes for the dominant filters.**
`price_history`: queries filter by `symbol` alone (`valuations.ts:66–71`, `assets.ts`,
`price-sync.ts:240`) but only a composite unique `(symbol, pricedDateUtc)`-style index exists —
fine if `symbol` is the leading column; verify, else add `index(symbol)`. `asset_valuations`: add
`(assetId, valuationDate)` covering index if not already leading-ordered for the
`lte(valuationDate)`+`desc` lookups in `report.ts:252` and overview.

**P5 — LOW-MEDIUM — `getTaxYears` loads every transaction and cash row.**
`server/tax/years.ts:5–6` selects all rows just to extract distinct years.
*Fix:* `SELECT min/max` (or distinct `strftime('%Y', ...)`) — two O(1) queries.

**P6 — LOW — `price-sync` FX lookup scans `fx_rates` per asset.**
`price-sync.ts:210–229` — fetch-all-then-filter per asset per currency. Prefetch one map per run.

**P7 — LOW — Misc.**
Top-positions sparkline payloads (~3,650 serialized points per page) `overview.ts:583–629`;
`audit_events` grows unbounded (reads are paginated — fine; optional retention later); Recharts in
four client components (acceptable; keep `lib/pdf/*` server-only — verified it is).

## 5. Regression Tests & Assertions for the Tax Invariant

- **R-1 Market-independence test (the invariant, executable):** seed transactions; build report;
  then `DELETE FROM price_history; DELETE FROM asset_valuations;` and assert
  `totals.*`, `sales[]`, `dividends[]` byte-identical, and `yearEndBalances` becomes *flagged
  unvalued* — never silently 0 (post-T4). Would have caught any future `?? marketPrice` regression.
- **R-2 Dividend FX guard:** `createDividend({currency:"USD"})` without `fxRateToEur` → expect
  `ok:false` validation error. (Today this test *fails* — it documents T1.)
- **R-3 Stale FX surfaced:** insert fx_rates row for D-7 only; `createTransaction` on D → row/audit
  carries `fxSource:"latest"`/stale marker; UI-level snapshot shows the warning.
- **R-4 Type-leak compile test:** `// @ts-expect-error` assigning `MarketEur` to a `TxEur` parameter
  (after T8) — makes the provenance wall a compiler guarantee.
- **R-5 Lot-integrity assertion (runtime + test):** for every sale, Σ`qtyConsumed` == sale qty and
  every `costBasisEur` ≥ 0; property test replaying random buy/sell sequences asserts
  Σ(lot costs consumed) + Σ(remaining lot costs) == Σ(buy gross+fees) within 1 cent.
- **R-6 Oversell atomicity:** import CSV with sell-before-buy → action returns `ok:false` and **zero**
  rows persisted in any table (transactions, lots, fx_rates, audit).
- **R-7 Sell-requires-price:** attempt sell with `priceNative: 0` / missing → rejected at schema;
  plus DB `CHECK` migration test.
- **R-8 Export formatting property test:** sum 1,000 amounts of €0.10 → casillas/detail CSV cells
  match `/^-?\d+\.\d{2}$/` exactly.
